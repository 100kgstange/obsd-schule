1. MQTT - Grundlagen #flashcard

MQTT (Message Queuing Telemetry Transport) ist ein leichtgewichtiges Protokoll für den Nachrichtenaustausch. Es wurde für Geräte mit geringer Bandbreite oder beschränkten Ressourcen entwickelt, wie IoT-Geräte.

1.1 Wichtige Eigenschaften von MQTT #flashcard

  • Protokoll: Client-Server (Broker-basiert).
  • Transport: Läuft üblicherweise über TCP (Port 1883) oder TCP+TLS (Port 8883).
  • Kommunikation: Publisher/Subscriber-Modell.
    - Publisher: Sendet Nachrichten an Themen (Topics).
    - Subscriber: Empfängt Nachrichten, indem er sich auf Themen abonniert.
    - Broker: Vermittler, der Nachrichten an die richtigen Subskriptionen weiterleitet (z. B. Mosquitto).

1.2 MQTT Topics #flashcard

  • Themen-Hierarchie: Hierarchische Struktur wie home/kitchen/temperature.
  • Wildcards:
    - +: Ein Platzhalter für eine Ebene (z. B. home/+/temperature).
    - #: Platzhalter für alle folgenden Ebenen (z. B. home/#).

1.3 Quality of Service (QoS) #flashcard

MQTT bietet drei QoS-Stufen, die die Zuverlässigkeit der Nachrichtenübermittlung definieren:

  1. QoS 0 (At most once — „Höchstens einmal“):
    • Die Nachricht wird einmal gesendet, ohne Bestätigung der Zustellung.
    • Es gibt keine Garantie, dass die Nachricht den Empfänger erreicht.
  2. QoS 1 (At least once — „Mindestens einmal“):
    • Die Nachricht wird mindestens einmal zugestellt.
    • Der Absender erhält eine Bestätigung über die Zustellung.
    • Es können Duplikate der Nachricht auftreten.
  3. QoS 2 (Exactly once — „Genau einmal“):
    - Die Nachricht wird genau einmal zugestellt.
    - Es gibt eine Garantie, dass keine Duplikate oder Verluste auftreten.

Anwendungsbeispiele

Hier sind Beispiele, in welchen Situationen die verschiedenen QoS-Level sinnvoll eingesetzt werden können:

QoS 0 (At most once — „Höchstens einmal“):

  • Beispiel 1: Sensordaten, die regelmäßig aktualisiert werden (z. B. Temperaturmesswerte).
    • Wenn ein Wert verloren geht, ist das nicht kritisch, da bald ein neuer Wert gesendet wird.
  • Beispiel 2: Live-Daten wie Wetterinformationen oder Aktienkurse.
    • Hier ist Echtzeit wichtiger als die Garantie, dass jede Nachricht ankommt.
  • Beispiel 3: Systeme mit begrenzten Ressourcen (z. B. Batteriebetriebene Geräte).
    • QoS 0 spart Energie, da keine Bestätigungen gesendet oder empfangen werden müssen.

QoS 1 (At least once — „Mindestens einmal“):

  • Beispiel 1: Steuerungsbefehle für Geräte (z. B. „Licht einschalten“).
    • Es ist wichtig, dass der Befehl ankommt, aber Duplikate sind nicht kritisch.
  • Beispiel 2: Alarmmeldungen (z. B. Rauchmelder oder Sicherheitssysteme).
    • Die Nachricht muss auf jeden Fall empfangen werden, auch wenn sie mehrmals gesendet wird.
  • Beispiel 3: Systeme, bei denen eine gewisse Zuverlässigkeit erforderlich ist, aber die Komplexität von QoS 2 vermieden werden soll.

QoS 2 (Exactly once — „Genau einmal“):

  • Beispiel 1: Finanztransaktionen (z. B. Zahlungsbestätigungen).
    • Es darf weder ein Verlust noch eine doppelte Zustellung der Nachricht geben.
  • Beispiel 2: Kritische Steuerungsbefehle (z. B. „Tür öffnen“ oder „Maschine stoppen“).
    • Hier ist es entscheidend, dass der Befehl genau einmal ausgeführt wird.
  • Beispiel 3: Systeme, bei denen Datenintegrität oberste Priorität hat (z. B. medizinische Geräte oder industrielle Steuerungen).

Zusammenfassung:

  • QoS 0: Geeignet für nicht-kritische Daten, bei denen Geschwindigkeit und Ressourceneffizienz wichtiger sind als Zuverlässigkeit.
  • QoS 1: Geeignet für Systeme, bei denen die Nachricht auf jeden Fall ankommen muss, aber Duplikate kein Problem darstellen.
  • QoS 2: Geeignet für kritische Anwendungen, bei denen Datenverlust oder Duplikate nicht akzeptabel sind.

2. Mosquitto - Der MQTT-Broker

Mosquitto ist eine beliebte Open-Source-Implementierung eines MQTT-Brokers.

2.1 Installation

  • Linux: sudo apt install mosquitto mosquitto-clients.
  • Mosquitto-Clients: Tools wie mosquitto_pub und mosquitto_sub.

2.2 Mosquitto Konfiguration #flashcard

  • Wichtige Einstellungen in mosquitto.conf:
    - Listener: Port und Adressen (Standard: 1883).
    - Authentifizierung:
    - Passwortdatei: password_file /etc/mosquitto/passwd.
    - Zugriff steuern: allow_anonymous true/false.
    - TLS-Sicherheit:
    - Zertifikate konfigurieren (cafile, certfile, keyfile).

2.3 Nützliche Befehle

  1. Nachrichten senden: #flashcard
    bash mosquitto_pub -h 192.168.0.34 -t test/topic -m "Hello MQTT"

  2. Nachrichten abonnieren: #flashcard
    bash mosquitto_sub -h 192.168.0.34 -t test/topic

3. Wichtige Begriffe und Konzepte

3.1 Client-Architektur

  • MQTT-Clients kommunizieren mit dem Broker, nie direkt miteinander.
  • Typische Clients: Sensoren, Aktoren, mobile Apps.

3.2 Retained Messages

  • Nachrichten werden vom Broker gespeichert, um neue Abonnenten sofort zu informieren.

3.3 Last Will and Testament (LWT)

  • Beim Verbindungsverlust kann der Broker eine vordefinierte Nachricht senden.

4. Sicherheit in MQTT #flashcard

Die Sicherheit eines Mosquitto-Brokers ist essenziell, da MQTT-Broker häufig sensible Daten zwischen Clients weiterleiten. Eine unsichere Konfiguration kann dazu führen, dass Unbefugte Nachrichten lesen oder manipulieren können.

  • TLS-Verschlüsselung: Aktivieren für sichere Verbindungen (cafile, certfile, keyfile).
  • Authentifizierung: Passwortdatei (password_file /etc/mosquitto/passwd) oder Zertifikate.
  • Themen-Berechtigungen: Themenzugriff granular einschränken.

  1. Broker einrichten: Installiere Mosquitto, starte den Broker, und veröffentliche eine Nachricht.
  2. Konfiguration:
    • Konfiguriere den Broker, um TLS zu verwenden.

  3. Themenmanagement: Abonniere ein Thema mit einem Platzhalter und veröffentliche Nachrichten auf verschiedenen Unterthemen.

Erklärung der mosquitto.conf - Datei #flashcard

# Place your local configuration in /etc/mosquitto/conf.d/
# A full description of the configuration file is at
# /usr/share/doc/mosquitto/examples/mosquitto.conf.example
#
# Definiert, dass Mosquitto auf Port 1883 (Standard-MQTT-Port) lauscht.
listener 1883
#
# Gibt die Datei für das Zertifikat der Zertifizierungsstelle (CA) an, das zur Überprüfung von SSL/TLS-Zertifikaten verwendet wird.
# 
cafile /ca.crt
# Der Pfad zum Server-Zertifikat für den MQTT-Broker.
certfile /broker.crt
#
# Der Pfad zur privaten Schlüsseldatei des Brokers.
keyfile /broker.key
# 
# Aktiviert die Überprüfung von Client-Zertifikaten.
require_certificate true
# 
# Erlaubt anonyme Verbindungen ohne Authentifizierung. In Kombination mit TLS ist das ungewöhnlich und potenziell unsicher.
allow_anonymous true
# 
#Gibt eine Datei an, in der Benutzer mit ihren Passwörtern gespeichert sind, falls Authentifizierung aktiviert ist.
pid_file /run/mosquitto/mosquitto.pid
# 
# Gibt die PID-Datei an, die die Prozess-ID des Mosquitto-Dienstes enthält.
password_file /etc/mosquitto/passwd
# 
# Aktiviert die Speicherung von Sitzungs- und Nachrichteninformationen auf der Festplatte.
persistence true
# 
# Legt das Verzeichnis fest, in dem Persistenzdaten gespeichert werden.
persistence_location /var/lib/mosquitto/
# 
# Speichert Log-Dateien im angegebenen Pfad.
log_dest file /var/log/mosquitto/mosquitto.log
# 
# Schließt zusätzliche Konfigurationsdateien aus dem angegebenen Verzeichnis ein.
include_dir /etc/mosquitto/conf.d

listener 1883 - standart MQTT Broker
oder
listener 8883 wenn TLS aktiv
cafile /ca.crt - Zertifikat von CA
certfile /broker.crt - Zertifikat vom Broker
keyfile /broker.key - Privatschlüssel vom Broker
require_certificate true - Aktiviert Überprüfung von Client-Zertifikaten
allow_anonymous true - erlaubt Verbindung ohne Authentifiizierung (bei TLS muss false sein)
password_file /etc/mosquitto/passwd - hier werden Benutzername und PW gesichert
pid_file /run/mosquitto/mosquitto.pid - Prozess-ID des Mosquitto-Dienstes
mosquitto_sub -h localhost -t test/topic
mosquitto_pub -h host -t test/topic -m "Hello"
